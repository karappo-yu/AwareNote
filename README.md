# AwareNote

AwareNote 是一个面向本地使用的资料浏览工具，主要解决 `PDF + 原生图片目录` 的管理与预览问题。

它不是面向公网的在线服务，也不是通用媒体库管理器。它更像一个为个人资料库准备的本地工具：尊重现有目录结构，只做索引、缓存和预览，不碰源文件。

## 为什么会有这个项目

这个项目诞生的原因很直接：现成方案都没有很好解决我的实际痛点。

- 很多图片管理软件喜欢把内容扁平化管理，破坏原本有意义的目录结构
- 很多漫画阅读器只偏向 `zip/cbz` 之类封装格式，对原生图片包支持不好
- 一些工具支持图片，但对 `PDF` 支持很弱
- 一些工具能看漫画，但没有适合资料浏览的网格封面视图
- 很多软件默认想“接管”你的文件，而不是尊重你已经整理好的目录

我的需求其实很简单：

- 目录本身就是组织方式
- 图片包本来就应该以文件夹的形式存在
- `PDF` 和图片目录都应该能统一管理
- 软件不应该改动、重命名、重组源文件

AwareNote 就是围绕这个边界做出来的。

## 解决了哪些痛点

- 支持同时管理 `PDF` 和原生图片目录
- 保留原始目录语义，不强制压缩、不强制导入库
- 通过封面缓存和页面缓存提升浏览体验
- 支持分类式浏览，而不是把所有内容打平成一个列表
- 支持本机和局域网设备访问，方便在 iPad、手机上浏览
- 对源文件零改动，只生成旁路缓存

## 项目原则

- 只解决真实存在的个人痛点，不追求“大而全”
- 不为了看起来通用而盲目支持一堆自己不用的格式
- 不动源文件，所有优化都走缓存
- 优先选择逻辑清晰、稳定可控的方案

这也是为什么项目当前明确聚焦在：

- `PDF`
- 原生图片目录

而没有去支持：

- `zip`
- `rar`
- `7z`
- 各类自动整理和自动改名逻辑

## 当前能力

- 扫描多个目录
- 自动识别书籍分类与子分类
- 支持图片书籍与 PDF 书籍
- 生成封面缓存
- 为大图图片包生成缩略预览缓存
- 为 PDF 页面生成 SVG 缓存
- 书籍详情页可直接在 Finder 中显示源文件位置
- 提供原生 macOS MenuBar App，用来启动服务和打开设置
- Web 端负责浏览和预览

这些能力大致对应下面几类界面：

- 资源库首页：封面网格、分类切换、资源浏览
- 详情页：封面、基础信息、路径、缓存状态
- 阅读页：单页预览与翻页浏览
- 原生设置页：服务、扫描、缓存等本地配置

## 不做什么

AwareNote 有一些明确不做的事情：

- 不改动源文件
- 不接管你的目录结构
- 不自动整理书籍
- 不强制导入数据库式媒体库
- 不优先支持你根本不用的压缩包格式
- 不把项目做成面向公网的复杂服务

## 适合什么场景

如果你的资料形态接近下面这样，这个项目就很适合：

- 很多 PDF
- 很多按目录整理好的图片包
- 希望保留 Finder 里的原始结构
- 需要封面网格浏览
- 偶尔希望在 iPad / 手机上访问本机资料库

## 运行方式

### 方式一：直接运行 Rust 后端

需要本机有 Rust 工具链。

```bash
cargo run --bin awarenotes
```

默认会根据配置文件启动本地服务，Web 界面通过浏览器访问。

### 方式二：打包成 macOS App

项目提供了 macOS 打包脚本：

```bash
./scripts/build-macos-app.sh
```

打包完成后，产物在：

```text
native-macos/dist/AwareNote.app
```

这个 App 是一个 macOS MenuBar App：

- 负责启动本地服务
- 提供原生设置入口
- Web 端仍然负责书籍浏览和预览

## 基本使用

1. 在配置中填写要扫描的目录
2. 启动服务
3. 在浏览器打开本机地址
4. 浏览分类、封面和详情页
5. 阅读时仍然通过 Web 页面进行预览

如果你使用的是打包版 macOS App，也可以通过菜单栏图标打开设置和服务入口。

## 配置说明

项目使用 `app_config.toml` 作为配置文件。

主要配置包括：

- 服务监听地址和端口
- 日志级别
- 扫描目录
- 图片数量阈值
- 缓存尺寸和渲染策略
- 内部并发参数

这个项目主要是本地自用工具，所以配置比普通面向用户的软件更直接一些。

## 技术实现

- 后端：Rust + Axum
- 数据库：SQLite
- ORM：SeaORM
- PDF 渲染：MuPDF
- 原生 macOS 菜单栏：`tao` + `tray-icon` + SwiftUI 设置窗口
- 前端：当前是旧版静态 Web 页面，后续如果重写会更偏向组件化方案

## 当前状态

这个项目已经基本完成了它最初的目标，后续大概率只做少量维护，不再持续扩张功能。

原因也很简单：

- 它已经解决了核心痛点
- 继续扩功能的收益不高
- 作为自用工具，维护成本比“功能完整”更重要

所以它更适合被理解为一个已经够用、边界明确的本地工具，而不是一个持续扩张的通用书库系统。
